## 1. 요약

CorpBrain은 **로컬 임베딩 + 하이브리드 검색 + Ollama LLM** 구조로, 사내 문서 외부 유출 없이 RAG 응답을 제공합니다. 본 보고서는 검색 품질·응답 지연·품질 게이트·확장성 관점의 성능을 정리합니다.

| 영역 | 현재 수준 | 목표 |
|------|-----------|------|
| 검색 Hit@3 | **62.5%** (8쿼리 eval) | **80%** |
| Hit@1 | 75.0% | — |
| MRR | 0.800 | — |
| 단위 테스트 | 44건 PASS | 유지 |
| E2E | 9건 PASS | 유지 |
| Quality Harness | PASS | CI 유지 |

> 검색 지표는 `vault` 인덱싱 후 `npm run eval:search` 기준입니다. 인덱스 미생성 시 Health `degraded`, metrics 0 반환.

## 2. 검색 품질 (Hit@K)

### 2.1 평가 방법

- 데이터: `data/eval-queries.json` (8개 질문, Role별)
- 파이프라인: `hybridSearch` (RRF + 한국어 리랭킹)
- 메트릭: `lib/search/metrics.ts` — Hit@1, Hit@3, MRR

```bash
npm run index:vault   # 선행 필수
npm run eval:search
```

### 2.2 집계 결과 (2026-07-02)

| 메트릭 | 값 | 비고 |
|--------|-----|------|
| Hit@1 | 75.0% | 6/8 |
| Hit@3 | 62.5% | 5/8 |
| MRR | 0.800 | — |
| 목표 Hit@3 | 80% | `targetHitAt3` (Admin UI) |
| CI 게이트 | ≥50% | `EVAL_HIT3_THRESHOLD` 기본값 |

### 2.3 개선 여지

| 항목 | 상태 | 기대 효과 |
|------|------|-----------|
| Cross-encoder 리랭킹 | 계획 (Phase 5) | Hit@3 ↑ |
| `ko-sroberta` A/B | 계획 | 한국어 의미 검색 ↑ |
| eval 쿼리 확대 | 8 → 20+ | 회귀 감지력 ↑ |
| 동의어 사전 확장 | 부분 적용 | 도메인 용어 매칭 ↑ |

## 3. 응답 지연 (Latency)

### 3.1 채팅 요청 구간

| 단계 | 예상 지연 | 병목 |
|------|-----------|------|
| 임베딩 생성 | 200~800ms | Transformers.js CPU, 최초 모델 로드 수 초 |
| hybridSearch | 10~100ms (JSON) / 5~50ms (PgVector ANN) | 저장소·청크 수 |
| Ollama TTFT | 1~5s | GPU/CPU, 모델 크기 |
| 스트리밍 전체 | 5~30s | `maxDuration=30` (Vercel 한도 대응) |

### 3.2 인덱싱

| 모드 | vault 22파일 기준 | 비고 |
|------|-------------------|------|
| full | ~1~3분 | 임베딩 per chunk |
| incremental | 변경 파일만 | manifest hash 비교 |

### 3.3 Admin Metrics API

- 전체 eval 재실행: 8쿼리 × (임베딩 + 검색) ≈ 수 초~수십 초
- **15분 캐시** (`CACHE_TTL_MS`) — `?refresh=true`로 강제 갱신

## 4. 처리량·제한

### 4.1 Rate Limit (인메모리)

| API | 한도 | 윈도우 |
|-----|------|--------|
| `/api/chat` | 20 req | 1분/사용자 |
| `/api/upload` | 10 req | 1분/사용자 |
| `/api/index` | 2 req | 1시간/사용자 |
| `/api/slack/command` | 30 req | 1분/Slack user |

초과 시 HTTP 429 + `Retry-After`

> 프로덕션 다중 인스턴스 환경에서는 Redis 기반 limiter 권장 (`lib/rate-limit.ts` 주석).

### 4.2 동시성

- Ollama: 단일 호스트 기준 동시 스트림 수 제한 — 운영 시 큐·워커 분리 검토
- Transformers.js: 요청별 순차 임베딩 — 배치 최적화 여지

## 5. 저장소 성능

### 5.1 JSON Vector Store (PoC)

| 항목 | 특성 |
|------|------|
| 적합 규모 | ~수천 청크 |
| 검색 | 메모리 로드 + 선형/하이브리드 스캔 |
| 장점 | 설정 없음, 로컬 PoC 빠른 시작 |
| 한계 | 대용량·다중 인스턴스 비적합 |

### 5.2 PgVector (운영)

| 항목 | 특성 |
|------|------|
| ANN | cosine `<=>` 인덱스 |
| ILIKE | 키워드 후보 보강 |
| 적합 규모 | 수만~수십만 청크 |
| 마이그레이션 | `npm run db:migrate` |
